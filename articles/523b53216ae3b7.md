## はじめに
エンジニアにとって、積み上げてきた「試行錯誤の履歴」を失うことは、何よりも恐ろしいことです。

私は、ファイル保存（Ctrl+S）のたびに裏側で超高頻度に履歴を残し、SourceTreeのような路線図から過去の保存点へ一発でタイムトラベルできるVS Code拡張機能「**MicroGit**」を開発しています。

「コミットメッセージを考える手間を省き、すべての作業を救い出す」

そんな理想のツールを作っていたはずの私が、開発の最終段階で直面したのは、**「自プロジェクトの`.git`フォルダが物理的に消滅する」**という、最悪のアイロニーに満ちた事故でした。

## 1. MicroGit（マイクロコミット構想）とは
開発の動機は、Gitでコミットする前の「ちょっとした試行錯誤」をすべて自動で記録したいという思いでした。

```text
 [保存(Ctrl+S)]  -->  [MicroGit]  -->  [自動コミット]
                                          |
 [任意地点へ復元] <--  [履歴ツリー表示] <---  [シャドウリポジトリ]
```

- **全自動シャドウコミット**: 保存をトリガーに、本体を汚さず `.microgit_shadow` 領域へ自動記録。
- **非線形な履歴管理**: 過去に戻って別の修正を始めたら、自動で「新しい世界線（mb-1, mb-2...）」として分岐。
- **直感的なUI**: 専用Webviewで、分岐したマイクロ履歴を視覚的に確認。

本家リポジトリの履歴を汚すことなく、ローカルでの試行錯誤をすべて「Undo Tree」として管理できるツールを目指していました。

## 2. 絶望の「Git消失事件」とその原因

v1.0.1のテスト中、異変は突然起きました。
VS Code上のファイルがすべて「未追跡（Untracked）」になり、あろうことか**プロジェクトの核である`.git`フォルダそのものが削除されてしまったのです。**

### なぜ実行ディレクトリが狂ったのか
調査の結果、Node.jsの `child_process` で指定する `cwd`（実行ディレクトリ）に起因する問題であることが分かりました。

当初、Windows環境でフォルダ名に「研究室関係」という日本語が含まれていたため、文字化けが原因だと推測していました。しかし真因は、**パス解決の不備でディレクトリ移動に失敗した際、実行場所が想定外の場所（メインプロジェクトのルート）に固定されてしまったこと**にありました。

### 暴発した破壊的コマンド
本来は、バックアップ領域であるシャドウリポジトリの掃除のために実行されるはずだった破壊的な操作が、移動の失敗によってメインプロジェクト側で実行されてしまいました。
```bash
# 本来は .microgit_shadow 内で実行されるはずが、ルートで暴発
git rm -rf --cached . 
```
これにより、インデックスが吹き飛ぶだけでなく、物理的な `.git` ディレクトリまでもが道連れになり、リポジトリが「ただのフォルダ」に成り果てたのです。

### 教訓：`cwd` に頼らず `-C` を使え
この失敗から学んだ鉄則は、Git操作を自動化する際はディレクトリ移動（`cwd`）に頼らず、Gitの **`-C "絶対パス"` オプション** を使うべきだということです。
`-C` を使えば、パスに日本語が含まれていようがいまいが、Gitが直接安全に対象リポジトリを解釈できるため、実行場所の取り違えを根本から防ぐことができます。

## 3. 復旧編：絶望から救ってくれたもの
自分の `.git` を消した瞬間は頭が真っ白になりました。しかし、私を救ったのは他ならぬ**「Gitそのもの」**でした。

幸いなことに、開発中のコードを定期的にGitHubへプッシュ（リモート同期）していたため、ネット上の履歴からプロジェクトを救出することができました。
```bash
git init
git remote add origin <URL>
git fetch
git reset --hard origin/master
```
おなじみの手順により、わずか数分で開発環境を元通りに再構築できたのです。

## おわりに：究極のアイロニー
今回の事故で最も痛感したのは、以下の事実です。

**「コミット回数を減らし、手間をなくすツールを作っていた本人が、こまめなコミットとPushによって救われた」**

どれほど便利な自動化ツールを作ろうとも、**「こまめなコミットこそがプロジェクトを救う最後の砦である」**というGitの原点を、自らの失敗によって思い知らされました。

この拡張機能「MicroGit」は、そんな泥臭いデバッグと、いくつもの「技術的な壁（非同期処理の同期、UIスタブの不安定さ、Windows環境でのパス問題）」を乗り越えて公開に至りました。

失敗の数だけ、ツールも私のGitへの理解も頑丈になりました。自分の `.git` を消しても、`git init` からやり直せば道は開けます。この教訓を胸に、これからも開発を続けていきます。
